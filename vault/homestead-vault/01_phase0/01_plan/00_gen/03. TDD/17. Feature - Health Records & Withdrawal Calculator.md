# Goal

Build the health records system with all 6 record types (vaccination, medication, deworming, vetVisit, illness, injury), dosage tracking, and the withdrawal period calculator. The withdrawal calculator is the **single highest-anxiety feature** and the **#1 subscription conversion driver**. Food safety depends on it.

# Instructions

## Architecture Notes

1. `HealthService` calls `@react-native-firebase/firestore` and `@react-native-firebase/storage` directly. All farm data lives under `/homestead/{homesteadId}/` subcollections. The service uses `this.homesteadRef` (a `DocumentReference` for `firestore().collection('homestead').doc(homesteadId)`) to access subcollections.
2. `admin.created_at` and `admin.updated_at` are `Tstamp`. Use `adminObject_default()` and `adminObject_updateLastUpdated()`.
3. Health records are historical data fetched on demand — no `onSnapshot` listener, use standard `get()` queries.
4. `HealthService` receives `ICareService` via constructor (injected in Bootstrap) to auto-create care reminders when a vaccination has a `vaccineNextDueDate`. This is the only constructor dependency.
5. No `userId` field on health record documents. No `userId` in queries or method signatures. The homestead subcollection path provides scoping.
6. Service location: `apps/mobile/src/feature/health/`.
7. Bootstrap export: `bsHealthService: IHealthService` from `apps/mobile/src/Bootstrap.ts`.

### `HealthService.ts`
```typescript
import firestore, { FirebaseFirestoreTypes } from '@react-native-firebase/firestore'
import storage from '@react-native-firebase/storage'
import { IResult, SuccessResult, ErrorResult } from '../../../util/Result'
import { HealthRecord } from '../../../schema/health/HealthRecord'
import { adminObject_default } from '../../../schema/object/AdminObject'
import ICareService from '../../care/service/ICareService'
import Log from '../../../library/log/Log'

export default class HealthService implements IHealthService {
  private careService: ICareService
  private homesteadRef: FirebaseFirestoreTypes.DocumentReference

  constructor(careService: ICareService) {
    this.careService = careService
    // homesteadRef is set during Bootstrap after homesteadId is known
    this.homesteadRef = null as any
  }

  async fetchHealthRecordsByAnimal(animalId: string): Promise<HealthRecord[]> {
    try {
      const snapshot = await this.homesteadRef
        .collection('healthRecord')
        .where('animalId', '==', animalId)
        .where('admin.deleted', '==', false)
        .orderBy('date', 'desc')
        .get()

      return snapshot.docs.map(doc => ({
        ...doc.data(),
        id: doc.id,
      } as HealthRecord))
    } catch (error: any) {
      Log.error('HealthService', 'fetchHealthRecordsByAnimal error: ' + error.message)
      return []
    }
  }

  async fetchAllWithdrawalRecords(): Promise<HealthRecord[]> {
    try {
      const snapshot = await this.homesteadRef
        .collection('healthRecord')
        .where('admin.deleted', '==', false)
        .where('withdrawalPeriodDays', '>', 0)
        .get()

      const medicationRecords = snapshot.docs.map(doc => ({
        ...doc.data(),
        id: doc.id,
      } as HealthRecord))

      // Also fetch deworming withdrawal records (separate query since
      // Firestore doesn't support OR on different fields)
      const dewormSnapshot = await this.homesteadRef
        .collection('healthRecord')
        .where('admin.deleted', '==', false)
        .where('dewormingWithdrawalDays', '>', 0)
        .get()

      const dewormRecords = dewormSnapshot.docs.map(doc => ({
        ...doc.data(),
        id: doc.id,
      } as HealthRecord))

      // Deduplicate (a record could have both medication and deworming withdrawals)
      const seen = new Set<string>()
      const combined: HealthRecord[] = []
      for (const r of [...medicationRecords, ...dewormRecords]) {
        if (!seen.has(r.id)) {
          seen.add(r.id)
          combined.push(r)
        }
      }
      return combined
    } catch (error: any) {
      Log.error('HealthService', 'fetchAllWithdrawalRecords error: ' + error.message)
      return []
    }
  }

  async createHealthRecord(record: HealthRecord, photoUri?: string): Promise<IResult> {
    try {
      const docRef = this.homesteadRef.collection('healthRecord').doc()
      record.id = docRef.id
      record.admin = adminObject_default()

      if (photoUri) {
        const homesteadId = this.homesteadRef.id
        const storagePath = `homestead/${homesteadId}/healthRecord/${record.id}/photo.jpg`
        await storage().ref(storagePath).putFile(photoUri)
        const downloadUrl = await storage().ref(storagePath).getDownloadURL()
        record.photoStorageRef = storagePath
        record.photoUrl = downloadUrl
      }

      await docRef.set(record)

      // Side effect: vaccination with nextDueDate auto-creates a care reminder
      if (record.recordType === 'vaccination' && record.vaccineNextDueDate) {
        await this.careService.createCareEvent({
          animalId: record.animalId,
          templateId: '',
          name: `Vaccination: ${record.name}`,
          type: 'careSingle',
          cycle: '',
          dueDate: record.vaccineNextDueDate,
          completedDate: '',
          contactName: record.providerName ?? '',
          contactPhone: record.providerPhone ?? '',
          notes: `Auto-created from vaccination record`,
          photoStorageRef: '',
          photoUrl: '',
          createdNextRecurringEvent: false,
        } as any)
      }

      return SuccessResult
    } catch (error: any) {
      Log.error('HealthService', 'createHealthRecord error: ' + error.message)
      return ErrorResult(error.message)
    }
  }
}
```

**Key implementation details:**
- All queries use `this.homesteadRef.collection('healthRecord')` — the singular `healthRecord` subcollection under `/homestead/{homesteadId}/`.
- No `userId` field on documents, no `userId` in queries. Scoping is provided by the homestead subcollection path.
- No `onSnapshot` — health records are historical data. Fetched on demand when viewing Animal Detail Health tab or withdrawal dashboard.
- `fetchAllWithdrawalRecords` runs two queries (medication withdrawal + deworming withdrawal) because Firestore doesn't support OR on different fields. Results are deduplicated client-side.
- `createHealthRecord` has a cross-service side effect: if the record is a vaccination with `vaccineNextDueDate`, it calls `this.careService.createCareEvent()` to auto-create a one-time care reminder. The `careService` is injected via constructor (wired in Bootstrap).
- Photo upload uses `@react-native-firebase/storage` directly. Storage path follows `homestead/${homesteadId}/healthRecord/${record.id}/photo.jpg`.

## HealthStore (Zustand)

```typescript
import { create } from 'zustand'
import { HealthRecord } from '../../schema/health/HealthRecord'
import { bsHealthService } from '../../Bootstrap'

interface HealthStoreState {
  healthRecords: HealthRecord[]
  withdrawalRecords: HealthRecord[]
  loading: boolean

  fetchHealthRecordsByAnimal: (animalId: string) => Promise<void>
  fetchAllWithdrawalRecords: () => Promise<void>
  clear: () => void
}

export const useHealthStore = create<HealthStoreState>((set) => ({
  healthRecords: [],
  withdrawalRecords: [],
  loading: false,

  fetchHealthRecordsByAnimal: async (animalId) => {
    set({ loading: true })
    const records = await bsHealthService.fetchHealthRecordsByAnimal(animalId)
    set({ healthRecords: records, loading: false })
  },

  fetchAllWithdrawalRecords: async () => {
    const records = await bsHealthService.fetchAllWithdrawalRecords()
    set({ withdrawalRecords: records })
  },

  clear: () => set({ healthRecords: [], withdrawalRecords: [], loading: false }),
}))
```

**Key difference from care/animal stores:** No `subscribe()` — health records use fetch-on-demand since they are historical data. The store has explicit `fetch` actions called from controllers.

The `HealthService` receives the `CareService` instance via constructor injection so it can auto-create care events when a vaccination has a next due date. This is wired in Bootstrap — the only place where cross-service dependencies are resolved.

# Key Design Decisions

1. **No hardcoded medication database** — farmers use off-label drugs frequently. The user enters withdrawal period manually.
2. **Multiple overlapping withdrawals** — one animal can have active meat withdrawal AND active milk withdrawal simultaneously.
3. **Withdrawal type matters** — an egg farmer with Corid (0-day) vs ivermectin (14+ day) has very different timelines.
4. **Dosage context always displayed together** — amount + unit + route + frequency are always shown as a group, never separated.
5. **Safety-critical UI** — active withdrawal status uses unmistakable red, cleared uses unmistakable green. No ambiguity.

# Subscription Gating

- **Free tier**: No access to health records. Attempting to create one triggers Pro paywall.
- **Pro/Farm tier**: Full access to all 6 record types and withdrawal calculator.

# Notes

- This is the #1 conversion driver — the moment a user tries to log a vaccination with a withdrawal period, the data is too valuable to abandon
- The withdrawal calculator surfaces on: Today Dashboard, Animal Detail banner, Health Record Detail
- Multiple overlapping withdrawals per animal are common (e.g., Penicillin with 4-day milk AND 8-day meat)
- Vaccination with `vaccineNextDueDate` creates a one-time care reminder — connects health records to the care system via cross-service call in Bootstrap
- Off-label use means the user determines the withdrawal period themselves — the app is intentionally flexible
- No `onSnapshot` needed — health records are historical data. Fetch on demand when viewing a specific animal's health tab or the withdrawal dashboard.
- Controllers never import Firebase — they call `bsHealthService` from Bootstrap and read from Zustand stores
- The `fetchAllWithdrawalRecords` method runs two Firestore queries (medication + deworming) and deduplicates client-side because Firestore doesn't support OR across different fields
- All data stored under `/homestead/{homesteadId}/healthRecord/` subcollection — no `userId` on documents
