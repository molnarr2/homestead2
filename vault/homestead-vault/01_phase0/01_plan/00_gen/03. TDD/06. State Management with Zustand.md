# Goal

Define the Zustand store architecture for Homestead. Each domain gets its own store. The homestead is the central data scope — stores subscribe to homestead subcollection data scoped by the active homestead ID. Stores call feature services (from Bootstrap), and real-time data flows through service `subscribe()` callbacks — stores never initiate Firebase listeners themselves.

# Instructions

## 1. Subscribe Pattern (store-holds-unsubscribe with teardown)

The core pattern for real-time stores. The store holds the unsubscribe function internally and provides `teardown()` to clean up:

```typescript
import { create } from 'zustand'
import { bsAnimalService } from '../Bootstrap'

interface AnimalState {
  animals: Animal[]
  loading: boolean
  unsubscribe: (() => void) | null
  subscribe: () => void
  teardown: () => void
  clear: () => void
}

const useAnimalStore = create<AnimalState>((set, get) => ({
  animals: [],
  loading: true,
  unsubscribe: null,

  subscribe: () => {
    get().teardown()
    set({ loading: true })
    const unsub = bsAnimalService.subscribe((animals) => {
      set({ animals, loading: false })
    })
    set({ unsubscribe: unsub })
  },

  teardown: () => {
    const unsub = get().unsubscribe
    if (unsub) unsub()
    set({ unsubscribe: null })
  },

  clear: () => {
    get().teardown()
    set({ animals: [], loading: true, unsubscribe: null })
  },
}))
```

The subscribe pattern:
1. Store exposes `subscribe()` which sets up a listener and stores the unsubscribe function internally
2. `subscribe()` calls `teardown()` first to clean up any existing listener
3. Service sets up `onSnapshot` and calls the callback on every change
4. Store updates its state in the callback
5. `teardown()` cleans up the Firestore listener
6. `clear()` calls `teardown()` and resets all state — used on sign-out

This pattern is used for ALL stores with real-time listeners.

## 2. Store Definitions

### `apps/mobile/src/store/authStore.ts` — Authentication State

```typescript
interface AuthState {
  isLoggedIn: boolean | null  // null = loading
  userId: string

  initialize: () => () => void  // returns unsubscribe
}

const useAuthStore = create<AuthState>((set) => ({
  isLoggedIn: null,
  userId: '',

  initialize: () => {
    const subscription = bsAuthService.loggedIn.subscribe({
      next: (loggedIn) => {
        set({
          isLoggedIn: loggedIn,
          userId: loggedIn ? bsAuthService.currentUserId : '',
        })
      },
    })
    return () => subscription.unsubscribe()
  },
}))
```

- Subscribes to `FirebaseAuth.loggedIn` (`BehaviorSubject<boolean>`) via rxjs `Subscribable`
- `isLoggedIn: null` means still loading — show splash screen
- `isLoggedIn: false` means show auth screens
- `isLoggedIn: true` means show main app
- No `isAnonymous` in store — read from `bsAuthService.isAnonymous` when needed

## 3. Which Stores Use Real-time vs Fetch-on-demand

| Store | Strategy | Reason |
|---|---|---|
| `authStore` | Real-time (rxjs) | Auth state changes must propagate instantly |
| `homesteadStore` | Real-time | Central data scope — holds active homestead ID for all farm data |
| `userStore` | Real-time (onSnapshot) | Profile updates should reflect immediately |
| `animalStore` | Real-time (onSnapshot) | Animal list is the core navigation — must be current |
| `careStore` | Real-time (onSnapshot) | Care events are time-sensitive, power the dashboard |
| `breedingStore` | Real-time for active (onSnapshot) | Active breedings shown on dashboard |
| `animalTypeStore` | Real-time (onSnapshot) | Used for species selection across many screens |
| `healthStore` | Fetch-on-demand | Historical per-animal data, not time-critical |
| `productionStore` | Fetch-on-demand | Historical reporting data |
| `noteStore` | Fetch-on-demand | Per-animal detail screen only |
| `weightStore` | Fetch-on-demand | Per-animal detail screen only |

## 4. Sign-out Flow

### `apps/mobile/src/store/resetAllStores.ts`

```typescript
import { useHomesteadStore } from './homesteadStore'
import { useUserStore } from './userStore'
import { useAnimalStore } from './animalStore'
import { useCareStore } from './careStore'
import { useHealthStore } from './healthStore'
import { useBreedingStore } from './breedingStore'
import { useProductionStore } from './productionStore'
import { useNoteStore } from './noteStore'
import { useWeightStore } from './weightStore'
import { useAnimalTypeStore } from './animalTypeStore'

export function resetAllStores() {
  useAnimalStore.getState().clear()
  useCareStore.getState().clear()
  useHealthStore.getState().clear()
  useBreedingStore.getState().clear()
  useProductionStore.getState().clear()
  useNoteStore.getState().clear()
  useWeightStore.getState().clear()
  useAnimalTypeStore.getState().clear()
  useUserStore.getState().clear()
  useHomesteadStore.getState().clear()
}
```

Sign-out sequence:
1. Call `resetAllStores()` — each store's `clear()` calls `teardown()` to unsubscribe its Firestore listener, then resets state
2. Call `bsAuthService.signout()`
3. Auth store's rxjs subscription fires `isLoggedIn: false` → navigation switches to auth screens

## 5. Store Conventions

- Each store uses `zustand/create` with the standard `(set, get) => ({})` pattern
- All stores with real-time listeners use the `subscribe/teardown/clear` pattern
- Store holds unsubscribe function internally in state (`unsubscribe: (() => void) | null`)
- `teardown()` cleans up the Firestore listener
- `clear()` calls `teardown()` and resets state
- `resetAllStores()` calls `clear()` on every store
- Async write actions return `Promise<IResult>` so screens can show success/error
- Stores are consumed in screens via hooks: `const { animals, loading } = useAnimalStore()`
- Stores are independent — no cross-store imports. Use Bootstrap services for shared logic.
- Loading state: `true` while waiting for initial data (either subscribe or fetch)

# Notes

- Zustand stores are lightweight and don't require providers or context wrappers
- The subscribe/teardown/clear pattern means each store manages its own Firestore listener lifecycle — no external unsubscribe tracking needed
- Unit test stores by mocking Bootstrap service exports
- `animalTypeStore` is the most complex — it manages the customization engine with nested subcollections
- Derived getters (like `getOverdueEvents`) filter the local array — they don't make additional Firestore queries
- If dashboard performance needs it, `healthStore.activeWithdrawals` could be upgraded to onSnapshot later
