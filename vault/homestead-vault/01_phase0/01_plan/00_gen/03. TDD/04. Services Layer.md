# Goal

Define the service layer architecture. Each feature domain gets its own service that calls `@react-native-firebase/*` directly. There are no plugin interfaces — services talk to Firebase, and UI/Controllers talk to services through Bootstrap.

Services access farm data through homestead subcollection paths. The active homestead ID comes from a centralized store, not from constructor injection.

The following library files support services but are not services themselves:
- `FirebaseAuth.ts` at `apps/mobile/src/library/cloudplatform/firebase/FirebaseAuth.ts` — the auth library used by AuthService
- `Log.ts` at `apps/mobile/src/library/log/Log.ts`
- `RevenueCatInAppPurchases.ts` at `apps/mobile/src/library/purchases/revenuecat/`

# Service Architecture Pattern

## Location
Each feature domain has a service that lives in:
```
apps/mobile/src/feature/{domain}/service/
  I{Domain}Service.ts           # Interface
  {Domain}Service.ts            # Implementation
```

## Pattern
```
┌────────────────┐     ┌──────────────┐     ┌──────────────────────┐
│  Screen (UI)   │────▶│  Controller  │────▶│  Zustand Store       │
│  Uniwind       │     │  (hook)      │     │  (state + actions)   │
└────────────────┘     └──────────────┘     └──────────┬───────────┘
                                                       │
                                                       ▼
                                            ┌──────────────────────┐
                                            │  Service             │
                                            │  (calls Firebase     │
                                            │   directly)          │
                                            └──────────┬───────────┘
                                                       │
                                                       ▼
                                            ┌──────────────────────┐
                                            │  @react-native-      │
                                            │  firebase/firestore  │
                                            │  firebase/storage    │
                                            └──────────────────────┘
```

**Critical rule:** Only services import `@react-native-firebase/*`. UI and Controllers NEVER import Firebase, MMKV, or other 3rd-party data libraries.

## Interface Contract

Every service interface follows this pattern:

```typescript
import { IResult } from '../../../util/Result'

export default interface I{Domain}Service {
  // Real-time listener (where applicable)
  subscribe{Collection}(callback: (items: T[]) => void): () => void

  // Read operations
  get{Entity}(id: string): Promise<T | null>

  // Mutations (always return IResult)
  create{Entity}(entity: T): Promise<IResult>
  update{Entity}(entity: T): Promise<IResult>
  delete{Entity}(id: string): Promise<IResult>
}
```

## Implementation Pattern

Farm data services access collections as subcollections under `/homestead/{homesteadId}/`. The `homesteadId` is read from a centralized Zustand store. No `userId` is passed to methods or stored on farm documents.

```typescript
import firestore from '@react-native-firebase/firestore'
import { IResult, SuccessResult, ErrorResult } from '../../../util/Result'
import { adminObject_updateLastUpdated } from '../../../schema/object/AdminObject'
import Log from '../../../library/log/Log'
import { useHomesteadStore } from '../../../store/homesteadStore'

const TAG = '{Domain}Service'

export default class {Domain}Service implements I{Domain}Service {

  private get homesteadRef() {
    const homesteadId = useHomesteadStore.getState().homesteadId
    return firestore().collection('homestead').doc(homesteadId)
  }

  // --- REAL-TIME LISTENER (onSnapshot) ---

  subscribe{Collection}(callback: (items: T[]) => void): () => void {
    return this.homesteadRef
      .collection('{collectionName}')
      .where('admin.deleted', '==', false)
      .onSnapshot(
        snapshot => {
          const items: T[] = snapshot.docs.map(doc => ({
            ...doc.data(),
            id: doc.id,
          } as T))
          callback(items)
        },
        error => {
          Log.error(TAG, `onSnapshot error: ${error.message}`)
          callback([])
        }
      )
  }

  // --- READ ---

  async get{Entity}(id: string): Promise<T | null> {
    try {
      const doc = await this.homesteadRef.collection('{collectionName}').doc(id).get()
      if (!doc.exists) return null
      return { ...doc.data(), id: doc.id } as T
    } catch (error: any) {
      Log.error(TAG, `get{Entity} error: ${error.message}`)
      return null
    }
  }

  // --- CREATE ---

  async create{Entity}(entity: T): Promise<IResult> {
    try {
      const ref = this.homesteadRef.collection('{collectionName}').doc()
      entity.id = ref.id
      await ref.set(entity)
      return SuccessResult
    } catch (error: any) {
      Log.error(TAG, `create{Entity} error: ${error.message}`)
      return ErrorResult(error.message)
    }
  }

  // --- UPDATE ---

  async update{Entity}(entity: T): Promise<IResult> {
    try {
      adminObject_updateLastUpdated(entity.admin)
      await this.homesteadRef.collection('{collectionName}').doc(entity.id).update(entity)
      return SuccessResult
    } catch (error: any) {
      Log.error(TAG, `update{Entity} error: ${error.message}`)
      return ErrorResult(error.message)
    }
  }

  // --- SOFT DELETE ---

  async delete{Entity}(id: string): Promise<IResult> {
    try {
      await this.homesteadRef.collection('{collectionName}').doc(id).update({
        'admin.deleted': true,
        'admin.updated_at': firestore.FieldValue.serverTimestamp(),
      })
      return SuccessResult
    } catch (error: any) {
      Log.error(TAG, `delete{Entity} error: ${error.message}`)
      return ErrorResult(error.message)
    }
  }
}
```

## 6. Breeding Service (`apps/mobile/src/feature/breeding/service/`)

### `IBreedingService.ts`

import { IResult } from '../../../util/Result'
import BreedingRecord from '../../../schema/breeding/BreedingRecord'

export default interface IBreedingService {
  subscribeActiveBreedings(callback: (records: BreedingRecord[]) => void): () => void

  createBreedingRecord(record: BreedingRecord): Promise<IResult>

  recordBirthOutcome(record: BreedingRecord, offspring: Animal[]): Promise<IResult>
  deleteBreedingRecord(id: string): Promise<IResult>

Uses `onSnapshot` for active breedings (status == 'active') via `this.homesteadRef.collection('breedingRecord')`. The `recordBirthOutcome` uses a **batch write** to update the breeding record and create offspring animal documents atomically:

```typescript
async recordBirthOutcome(record: BreedingRecord, offspring: Animal[]): Promise<IResult> {
  try {
    const batch = firestore().batch()

    // Update breeding record
    const recordRef = this.homesteadRef.collection('breedingRecord').doc(record.id)
    batch.update(recordRef, record as any)

    // Create offspring documents
    for (const animal of offspring) {
      const animalRef = this.homesteadRef.collection('animal').doc()
      animal.id = animalRef.id
      batch.set(animalRef, animal as any)
    }

    await batch.commit()
    return SuccessResult
  } catch (error: any) {
    Log.error(TAG, `recordBirthOutcome error: ${error.message}`)
    return ErrorResult(error.message)
  }
}
```

## 13. OnSnapshot — Where It's Used

Real-time listeners via `onSnapshot` are used on collections where data changes are time-sensitive and should reflect immediately in the UI:

| Subcollection | Service Method | Why Real-Time |
|-----------|---------------|---------------|
| `animal` | `AnimalService.subscribeAnimals()` | Animal list is the primary data view, updated from multiple screens |
| `careEvent` | `CareService.subscribeCareEvents()` | Care due dates change daily, completions should reflect instantly |
| `breedingRecord` (active only) | `BreedingService.subscribeActiveBreedings()` | Gestation countdowns are time-sensitive, shown on dashboard |
| `healthRecord` (active withdrawals) | `HealthService.subscribeActiveWithdrawals()` | Withdrawal alerts are critical safety information |

All of the above are subcollections under `/homestead/{homesteadId}/`.

**Collections that do NOT use onSnapshot** (standard async queries instead):
- `productionLog` — historical data, queried on demand
- `note` — historical data, queried per-animal
- `weightLog` — historical data, queried per-animal
- `animalType` — configuration data, rarely changes
- `feedback` — write-only from client (root collection, not under homestead)

### OnSnapshot Lifecycle Pattern

Services return an **unsubscribe function**. The Zustand store (or controller) is responsible for calling it on cleanup:

```typescript
// In Zustand store:
subscribeToAnimals: () => {
  const unsubscribe = bsAnimalService.subscribeAnimals((animals) => {
    set({ animals, loading: false })
  })
  set({ unsubscribe })
}

cleanup: () => {
  const { unsubscribe } = get()
  if (unsubscribe) unsubscribe()
}
```

Bootstrap notes:
- Most services take no constructor args (minimal injection). They get `homesteadId` from `useHomesteadStore.getState()` internally.
- Only services with cross-service side effects receive deps via constructor: `HealthService` receives `ICareService` for vaccination-to-care-event creation. `AuthService` takes `FirebaseAuth`, and `SubscriptionService` takes `RevenueCatInAppPurchases`.
- Every export is typed to its interface (`I{Domain}Service`), not the concrete class.
- Caching is handled within individual services or stores; there is no separate cache service.

## 16. Photo Upload Pattern

Services that support photo uploads follow this pattern:

```typescript
import storage from '@react-native-firebase/storage'

async uploadPhoto(path: string, localUri: string): Promise<{ url: string, ref: string } | null> {
  try {
    const ref = storage().ref(path)
    await ref.putFile(localUri)
    const url = await ref.getDownloadURL()
    return { url, ref: path }
  } catch (error: any) {
    Log.error(TAG, `uploadPhoto error: ${error.message}`)
    return null
  }
}
```

Storage paths are scoped to homestead for farm data, and to user for user-specific data:
- Animal photos: `homestead/{homesteadId}/animal/{animalId}/{filename}`
- Note photos: `homestead/{homesteadId}/note/{noteId}/{filename}`
- Health record photos: `homestead/{homesteadId}/healthRecord/{recordId}/{filename}`
- User avatars: `user/{userId}/avatar/{filename}` (user-scoped, not under homestead)

# Notes

- Services are the **ONLY** files that import `@react-native-firebase/firestore`, `@react-native-firebase/storage`, etc. (exception: `FirebaseAuth.ts`)
- UI and Controllers import from `Bootstrap.ts` — never from Firebase directly
- All Firestore queries filter `admin.deleted == false` unless explicitly stated otherwise
- AuthService is a thin wrapper at `core/service/auth/` — it only delegates to FirebaseAuth, NO user document CRUD
- User document CRUD is handled by UserService at `feature/user/service/`
- All farm data services access collections through homestead subcollection paths: `/homestead/{homesteadId}/{collection}/`
- Services get `homesteadId` from `useHomesteadStore.getState().homesteadId` — not from constructor injection or method parameters
- No `userId` is stored on farm documents or passed to farm data service methods — ownership is implicit through the homestead
- Most services have no constructor deps (minimal injection). Only HealthService receives `ICareService` for cross-service vaccination-to-care-event side effects
- Every service has Interface + Implementation files (`I{Domain}Service.ts` + `{Domain}Service.ts`). Bootstrap exports are typed to interfaces
- Services use `try/catch` with `Log.error()` for error handling and return `IResult` for all mutations
- The `onSnapshot` unsubscribe pattern is critical to prevent memory leaks — stores must call unsubscribe on cleanup
- Batch writes are used when multiple documents need to be created/updated atomically (e.g., birth outcome recording)
- Storage paths: farm data under `homestead/{homesteadId}/`, user data under `user/{userId}/`
- Collection names are singular without `_v2` suffix: `animal`, `careEvent`, `breedingRecord`, `healthRecord`, `productionLog`, `note`, `weightLog`, `animalType`
