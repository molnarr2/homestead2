# Goal

Define all TypeScript schema interfaces in the `apps/mobile/src/schema/` directory for data models used across the app. All schemas include `admin: AdminObject` for soft-delete and timestamp tracking. Models are split into separate files per entity.

**Architecture decisions:**
- `AdminObject` uses `Tstamp` (Firebase Timestamps), not ISO strings
- No `SchemaDefinition.ts`, no `SchemaType` enum, no `SchemaRef`, no `CloudDocumentData`. Services reference Firestore collections directly by name.
- No `_get()`, `_set()`, `_collection()` helpers on any schema — services handle all Firestore access
- Each schema file contains **only**: the interface + `{entity}_default()` factory function

## 4. Built-in Gestation Table

### `apps/mobile/src/schema/type/GestationTable.ts`

```typescript
export const GESTATION_TABLE: Record<string, number> = {
  'Chicken': 21,
  'Duck': 28,
  'Turkey': 28,
  'Goose': 30,
  'Quail': 17,
  'Rabbit': 31,
  'Pig': 114,
  'Sheep': 147,
  'Goat': 150,
  'Cattle': 283,
  'Horse': 340,
  'Alpaca': 345,
  'Llama': 350,
  'Donkey': 365,
}

export const POULTRY_TYPES = ['Chicken', 'Duck', 'Turkey', 'Goose', 'Quail']

export function getGestationDays(animalType: string): number {
  return GESTATION_TABLE[animalType] ?? 0
}

export function isPoultry(animalType: string): boolean {
  return POULTRY_TYPES.includes(animalType)
}

export function getGestationLabel(animalType: string): string {
  return isPoultry(animalType) ? 'incubation' : 'gestation'
}
```

# Firestore Collection Names

Services reference collections directly by string name. Here is the canonical list of Firestore collection paths:

| Collection | Path | Ownership |
|-----------|------|-----------|
| User | `user/{userId}` | doc ID = auth UID |
| UserDevice | `user/{userId}/device/{deviceId}` | subcollection of User |
| Homestead | `homestead/{homesteadId}` | membership via member subcollection |
| HomesteadMember | `homestead/{homesteadId}/member/{memberId}` | subcollection of Homestead |
| Animal | `homestead/{homesteadId}/animal/{animalId}` | subcollection of Homestead |
| CareEvent | `homestead/{homesteadId}/careEvent/{eventId}` | subcollection of Homestead |
| HealthRecord | `homestead/{homesteadId}/healthRecord/{recordId}` | subcollection of Homestead |
| BreedingRecord | `homestead/{homesteadId}/breedingRecord/{recordId}` | subcollection of Homestead |
| Note | `homestead/{homesteadId}/note/{noteId}` | subcollection of Homestead |
| WeightLog | `homestead/{homesteadId}/weightLog/{logId}` | subcollection of Homestead |
| ProductionLog | `homestead/{homesteadId}/productionLog/{logId}` | subcollection of Homestead |
| AnimalType | `homestead/{homesteadId}/animalType/{typeId}` | subcollection of Homestead |
| Breed | `homestead/{homesteadId}/animalType/{typeId}/breed/{breedId}` | subcollection of AnimalType |
| CareTemplate | `homestead/{homesteadId}/animalType/{typeId}/careTemplate/{templateId}` | subcollection of AnimalType |
| EventTemplate | `homestead/{homesteadId}/animalType/{typeId}/eventTemplate/{templateId}` | subcollection of AnimalType |
| Feedback | `feedback/{feedbackId}` | `userId` field |

# Notes

- **Homestead is the central entity.** All farm data lives under `/homestead/{homesteadId}/`. Users access farm data through homestead membership. User profiles and feedback remain at root level.
- **No `_v2` suffix** on homestead subcollections — they live under a different root path (`/homestead/`) so there's no collision with existing production data. The `user` collection is still shared (with `schemaVersion` field).
- **Production data coexistence:** The `user` collection is reused (with `schemaVersion` field). Farm data under `/homestead/` is in a separate namespace from the old app's flat collections.
- **User.schemaVersion** — set to `2` on new user documents. Allows distinguishing v2 users from legacy production users in the shared `user` collection.
- **Every schema has `admin: AdminObject`** — provides `deleted`, `updated_at`, `created_at` with Firebase Timestamps
- **AdminObject timestamps are `Tstamp` (Firebase Timestamps)**, NOT ISO strings. Use `tstampServerTime()` for defaults.
- **`dueDate` and `completedDate` on CareEvent are `Tstamp`** (Firebase Timestamps), NOT ISO strings. This enables native Firestore range queries and ordering. All other domain dates (birthday, breedingDate, etc.) remain ISO 8601 strings.
- **Domain dates** (birthday, breedingDate, etc.) are **ISO 8601 strings**. Use `date-fns` for parsing and formatting.
- `AnimalType` has subcollections: `breed`, `careTemplate`, `eventTemplate`
- The `User.devices` subcollection maps to `/user/{userId}/device/{deviceId}` for FCM tokens
- Animal `sireId` and `damId` are self-referencing — they point to other animal documents
- HealthRecord uses a union-type pattern: `recordType` determines which conditional fields are populated
- ProductionLog uses either `animalId` OR `groupName`, never both
- `cycle: number` is always a number (days between recurring events), never a string
- Services handle Firestore access with `@react-native-firebase/firestore` directly — no `_get()`, `_set()`, or `_collection()` helpers on schemas
