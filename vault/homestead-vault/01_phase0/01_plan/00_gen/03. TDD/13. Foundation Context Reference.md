## Architecture Summary

- **Monorepo**: All mobile code lives under `apps/mobile/src/`. Not `HomesteadApp/src/`.
- **Homestead is the root entity.** All farm data collections live under `/homestead/{homesteadId}/`. Users access farm data through homestead membership (role-based). User profiles and feedback remain at root level. Farm data is scoped by homestead path — no `userId` field on farm documents.
- **Homestead membership** — Users are members of homesteads with roles (`owner`, `manager`, `caretaker`, `viewer`). Any member can read/write farm data. Security rules check membership via the `member` subcollection.
- **AdminObject** on every schema: `{ deleted: boolean, updated_at: Tstamp, created_at: Tstamp }`. Uses Firebase `Timestamp` / `FieldValue` (via `Tstamp` type), NOT ISO strings. `adminObject_default()` sets server timestamps.
- **Tstamp** type = `FirebaseFirestoreTypes.Timestamp | FieldValue`. Conversion helpers: `tstampToDate()`, `tstampToDateOrNow()`, `tstampServerTime()`, `dateToTstamp()`. All exist in `apps/mobile/src/schema/type/Tstamp.ts`.
- **Result** type: `IResult = { success: boolean, error: string }`. `SuccessResult` and `ErrorResult()` factory. Exists in `apps/mobile/src/util/Result.ts`.
- **State management**: Zustand stores. Each domain has its own store. No rxjs in stores (rxjs is only in FirebaseAuth).
- **UI**: Uniwind (Tailwind CSS for RN, drop-in NativeWind replacement). No react-native-paper.
- **Navigation**: Root stack → Drawer (side menu) → Bottom tabs (4 tabs: Home, Animals, Production, Care). Side menu accessed via avatar top-left.
- **Controller pattern**: `{Feature}Controller.ts` (hook) + `{Feature}Screen.tsx` (UI). Controllers use stores and Bootstrap services.
- **Bootstrap.ts**: Singleton DI — instantiates services and exports them as `bs{ServiceName}`.
- **No plugin interfaces**: Services call `@react-native-firebase/*` directly.
- **AuthService is a thin wrapper** around `FirebaseAuth`, which wraps `@react-native-firebase/auth` behind an `IAuthService` interface with rxjs `BehaviorSubject` for auth state. All other Firebase modules (Firestore, Storage, Functions) are called directly in services.
- **Services are per-feature**: Each feature has its own service(s) in `apps/mobile/src/feature/{name}/service/`. Services house ALL 3rd-party library code — UI and Controllers NEVER import Firebase.
- **OnSnapshot for real-time data**: Where it makes sense (animals list, care events, active breedings), services use `firestore().collection(...).onSnapshot()` and push updates into Zustand stores.
- **No SchemaRef helpers**: Services use Firestore directly via homestead path: `firestore().collection('homestead').doc(homesteadId).collection('animal').get()`.
- **Subscription gating**: RevenueCat client-side enforcement. Three tiers: Free, Pro, Farm.
- **Log**: Singleton logger (`Log.error()`, `Log.info()`, `Log.debug()`) that writes to console + Crashlytics.

## Common Patterns for Feature Implementation

When building a new feature:

### 1. Schema
Create model file(s) in `apps/mobile/src/schema/{domain}/` with:
- TypeScript interface
- Default factory function (`{name}_default()`)
- Include `admin: AdminObject`
- **No** `_get`, `_set`, `_collection` helpers — services handle Firestore access

```typescript
// apps/mobile/src/schema/animal/Animal.ts
import AdminObject, { adminObject_default } from '../object/AdminObject'

export default interface Animal {
  id: string
  name: string
  animalTypeId: string
  breed: string
  birthday: string  // ISO 8601
  // ... other fields
  admin: AdminObject
}

export function animal_default(): Animal {
  return {
    id: '',
    name: '',
    animalTypeId: '',
    breed: '',
    birthday: '',
    admin: adminObject_default(),
  }
}
```

### 2. Service
Create feature service in `apps/mobile/src/feature/{name}/service/`. Service calls Firestore directly. May use `onSnapshot` for real-time collections.

```typescript
// apps/mobile/src/feature/animal/service/AnimalService.ts
import firestore from '@react-native-firebase/firestore'
import Animal from '../../../schema/animal/Animal'

export default class AnimalService {
  private unsubscribe: (() => void) | null = null

  subscribe(homesteadId: string, onUpdate: (animals: Animal[]) => void): void {
    const homesteadRef = firestore().collection('homestead').doc(homesteadId)
    this.unsubscribe = homesteadRef
      .collection('animal')
      .where('admin.deleted', '==', false)
      .onSnapshot(snapshot => {
        const animals = snapshot.docs.map(doc => ({ id: doc.id, ...doc.data() } as Animal))
        onUpdate(animals)
      })
  }

  teardown(): void {
    this.unsubscribe?.()
    this.unsubscribe = null
  }

  async createAnimal(homesteadId: string, animal: Animal): Promise<string> {
    const homesteadRef = firestore().collection('homestead').doc(homesteadId)
    const ref = homesteadRef.collection('animal').doc()
    await ref.set({ ...animal, id: ref.id })
    return ref.id
  }
}
```

### 3. Store
Create Zustand store in `apps/mobile/src/store/{domain}Store.ts`. Store calls service methods from Bootstrap. Holds domain state.

```typescript
// apps/mobile/src/store/animalStore.ts
import { create } from 'zustand'
import { bsAnimalService } from '../Bootstrap'
import Animal from '../schema/animal/Animal'

interface AnimalState {
  animals: Animal[]
  loading: boolean
  subscribe: (homesteadId: string) => void
  teardown: () => void
  clear: () => void
}

export const useAnimalStore = create<AnimalState>((set, get) => ({
  animals: [],
  loading: true,

  subscribe: (homesteadId: string) => {
    bsAnimalService.subscribe(homesteadId, (animals) => {
      set({ animals, loading: false })
    })
  },

  teardown: () => {
    bsAnimalService.teardown()
  },

  clear: () => {
    get().teardown()
    set({ animals: [], loading: true })
  },
}))
```

### 4. Controller
Custom hook in `apps/mobile/src/feature/{name}/{Feature}Controller.ts`. Uses Zustand stores. Returns data + actions for the screen.

```typescript
export function useAnimalListController(navigation: any) {
  const animals = useAnimalStore(s => s.animals)
  const loading = useAnimalStore(s => s.loading)

  const onAnimalPress = (animalId: string) =>
    navigation.navigate('AnimalDetail', { animalId })

  const onCreatePress = () =>
    navigation.navigate('CreateAnimal', {})

  return { animals, loading, onAnimalPress, onCreatePress }
}
```

### 5. Screen
UI component in `apps/mobile/src/feature/{name}/{Feature}Screen.tsx`. Uses controller hook. Imports shared components from `apps/mobile/src/components/`.

```typescript
const AnimalListScreen: React.FC = () => {
  const navigation = useNavigation()
  const { animals, loading, onAnimalPress, onCreatePress } = useAnimalListController(navigation)

  if (loading) return <LoadingIndicator />

  return (
    <ScreenContainer>
      <FlatList
        data={animals}
        renderItem={({ item }) => <AnimalCard animal={item} onPress={() => onAnimalPress(item.id)} />}
        ListEmptyComponent={<EmptyState icon="paw" title="No animals yet" subtitle="Tap + to add your first animal" />}
      />
      <FloatingActionButton onPress={onCreatePress} />
    </ScreenContainer>
  )
}
```

### 6. Navigation
Add route to `RootStackParamList` (detail/create screens) or `MainTabParamList` (tab screens) in `apps/mobile/src/navigation/RootNavigation.tsx`.

# Notes

- Feature docs (14+) should reference this document instead of repeating the full architecture
- When a feature needs a new Zustand store, add it to `resetAllStores()` in `apps/mobile/src/store/resetAllStores.ts`
- All Firestore queries filter `admin.deleted == false` unless explicitly stated otherwise
- All farm document creation uses the homestead path — no `userId` field on farm documents. Services receive `homesteadId` from the store.
- Photo uploads go to Firebase Storage under `homestead/{homesteadId}/animal/{animalId}/` or `user/{userId}/avatar/` paths — services call `@react-native-firebase/storage` directly
- OnSnapshot listeners must be cleaned up: services hold unsubscribe internally and expose `teardown()` methods, stores call `teardown()` in their `clear()` method, `resetAllStores()` calls `clear()` on each store
- Use `date-fns` for all date math — convert `Tstamp` → `Date` via `tstampToDate()` before passing to date-fns
