# Goal

Define all TypeScript schema interfaces in the `src/schema/` directory for data models used across the React Native app. All schemas include the `AdminObject` for soft-delete and timestamp tracking. Models are split into separate files per entity — no monolithic `models.ts`.

**Data architecture:** All farm data lives under `/homestead/{homesteadId}/` as subcollections. A homestead is the central organizing entity — users are members of homesteads, and all animals, care events, health records, etc. belong to a homestead. This supports multi-user access (inviting a caretaker, sharing a farm). User profiles and feedback remain at root level.

## 3. Schema Models — One File Per Entity

Each file follows this pattern: interface, default factory, `_get()`, `_set()`, `_collection()` helpers.
Homestead-scoped schemas pass `homesteadId` to their path helpers. No `userId` field on homestead-scoped docs — ownership is determined by homestead membership.

## 4. Built-in Gestation Table

Create `src/schema/type/GestationTable.ts`:

```typescript
export const GESTATION_TABLE: Record<string, number> = {
  'Chicken': 21,
  'Duck': 28,
  'Turkey': 28,
  'Goose': 30,
  'Quail': 17,
  'Rabbit': 31,
  'Pig': 114,
  'Sheep': 147,
  'Goat': 150,
  'Cattle': 283,
  'Horse': 340,
  'Alpaca': 345,
  'Llama': 350,
  'Donkey': 365,
}

export const POULTRY_TYPES = ['Chicken', 'Duck', 'Turkey', 'Goose', 'Quail']

export function getGestationDays(animalType: string): number {
  return GESTATION_TABLE[animalType] ?? 0
}

export function isPoultry(animalType: string): boolean {
  return POULTRY_TYPES.includes(animalType)
}

export function getGestationLabel(animalType: string): string {
  return isPoultry(animalType) ? 'incubation' : 'gestation'
}
```

# Notes

- **Every schema has `admin: AdminObject`** — provides `deleted`, `updated_at`, `created_at`
- All dates are ISO 8601 strings. Use `date-fns` for parsing and formatting.
- **Homestead-scoped architecture:** All farm data lives under `/homestead/{homesteadId}/` as subcollections. No `userId` field on farm data docs — access is controlled by homestead membership.
- **User-level data** (`user`, `feedback`) stays at root level — not homestead-specific
- `User.homesteadIds[]` provides quick lookup of which homesteads a user belongs to. `User.activeHomesteadId` tracks the currently selected homestead.
- `selectedSpecies` moved from User to Homestead onboarding flow (species are per-homestead, not per-user)
- `AnimalType` has subcollections: `breed`, `careTemplate`, `eventTemplate` — path helpers take `homesteadId` + `animalTypeId`
- The `User.devices` subcollection maps to `/user/{userId}/device/{deviceId}` for FCM tokens
- Animal `sireId` and `damId` are self-referencing — they point to other animal documents within the same homestead
- HealthRecord uses a union-type pattern: `recordType` determines which conditional fields are populated
- ProductionLog uses either `animalId` OR `groupName`, never both
- **Multi-user access:** Users are added as members of a homestead with a role (`owner`, `manager`, `caretaker`, `viewer`). Security rules check membership via the `member` subcollection.
