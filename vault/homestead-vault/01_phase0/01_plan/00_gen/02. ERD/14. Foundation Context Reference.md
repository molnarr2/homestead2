# Architecture Summary

- **User is the root entity.** All data collections use `userId` for ownership. No ministry/church hierarchy — this is a single-user (or farm-level) app.
- **AdminObject** on every schema: `{ deleted, updated_at, created_at }` for soft-delete and timestamps.
- **All dates** stored as ISO 8601 strings. Use `date-fns` for all date calculations.
- **State management**: Zustand stores (no rxjs). Each domain has its own store.
- **UI**: Uniwind (Tailwind CSS for RN, drop-in NativeWind replacement). No react-native-paper.
- **Navigation**: Root stack → Drawer (side menu) → Bottom tabs (4 tabs: Home, Animals, Production, Care). Side menu accessed via avatar top-left.
- **Controller pattern**: `{Feature}Controller.ts` (hook) + `{Feature}Screen.tsx` (UI). Controllers use stores and Bootstrap services.
- **Bootstrap.ts**: Singleton DI — instantiates services and exports them as `bs{ServiceName}`.
- **Plugin interfaces**: Abstract third-party libraries behind `ICloudPlatform`, `ICloudPlatformAuth`, `ILocalStorage`, etc.
- **Subscription gating**: RevenueCat client-side enforcement. Three tiers: Free, Pro, Farm.

# Common Patterns for Feature Implementation

When building a new feature:

1. **Schema**: Create model file(s) in `src/schema/{domain}/` with interface, default factory, `_get`, `_set`, `_collection` helpers. Include `admin: AdminObject`.
2. **Store**: Create Zustand store in `src/store/{domain}Store.ts` with state, fetch actions, mutation actions, and `clear()`.
3. **Service** (if needed): Create interface in `src/feature/{name}/service/`, implementation delegates to Bootstrap plugins.
4. **Controller**: Custom hook in `src/feature/{name}/{Feature}Controller.ts`. Uses stores and services. Returns data + actions for the screen.
5. **Screen**: UI component in `src/feature/{name}/{Feature}Screen.tsx`. Uses controller hook. Imports shared components from `src/components/`.
6. **Navigation**: Add route to `RootStackParamList` if it's a push screen, or to `MainTabParamList` if it's a tab.

# Notes

- Feature docs below (15+) should reference this document instead of repeating the full architecture.
- When a feature needs a new Zustand store, add it to `resetAllStores()` in `src/store/resetAllStores.ts`.
- All Firestore queries filter `admin.deleted == false` unless explicitly stated otherwise.
- All document creation must set `userId` from `useAuthStore().userId` or `bsAuthService.getUserId()`.
- Photo uploads go to Firebase Storage under `users/{userId}/` or `animals/{animalId}/` paths.
