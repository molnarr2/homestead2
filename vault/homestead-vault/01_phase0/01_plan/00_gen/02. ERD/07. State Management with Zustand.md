# Goal

Define the Zustand store architecture for Homestead. Zustand replaces rxjs-based reactive patterns with a simpler, hook-based state management approach. Each domain gets its own store slice.

## Store Architecture

### `src/store/authStore.ts` — Authentication State
```typescript
interface AuthState {
  isLoggedIn: boolean | null  // null = loading
  userId: string
  isAnonymous: boolean

  setLoggedIn: (loggedIn: boolean) => void
  setUserId: (id: string) => void
  initialize: () => () => void  // returns unsubscribe
}
```
- On `initialize()`, subscribe to `bsAuthService.onAuthStateChanged()` and update store
- Screens read `isLoggedIn` to determine nav state (Loading → Auth → Main)

## Store Convention

- Each store uses `zustand/create` with the standard pattern
- Async actions call Bootstrap services (`bsAuthService`, `bsCacheService`, etc.) or `ICloudPlatform` directly
- All stores export a `clear()` method called on sign-out
- Stores are consumed in screens via hooks: `const { animals, fetchAnimals } = useAnimalStore()`

## Sign-out Flow

Create a `src/store/resetAllStores.ts` utility:
```typescript
export function resetAllStores() {
  useUserStore.getState().clear()
  useAnimalStore.getState().clear()
  useCareStore.getState().clear()
  useHealthStore.getState().clear()
  useBreedingStore.getState().clear()
  useProductionStore.getState().clear()
  useNoteStore.getState().clear()
  useWeightStore.getState().clear()
  useAnimalTypeStore.getState().clear()
}
```

# Notes

- Zustand stores are lightweight and don't require providers or context
- Each store is independent — no cross-store dependencies (use Bootstrap services for shared logic)
- Stores handle loading/error states internally
- Unit test stores by mocking Bootstrap service exports
- The `animalTypeStore` is the most complex — it manages the customization engine with nested subcollections
